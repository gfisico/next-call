# バージョン番号ルール（汎用・全アプリ展開テンプレート）

> Iconizer で運用しているバージョン番号ルールを、他アプリへ流用できるよう汎用化したもの。
> このファイルをコピーして、各アプリの要約（CLAUDE.md / 設計ドキュメント等）に組み込む。

---

## 1. SSOT（唯一の正）

バージョン番号の唯一の正は **単一ファイル** とする。他の場所に同じ値を持つ二重管理をしない。

| 対象 | SSOT | 補足 |
|---|---|---|
| バージョン番号 | `src/version.ts`（アプリ構成に合わせて配置） | 定数を1つだけ export する |

参考実装（TypeScript）:

```ts
// VERSION SSOT — 更新ルール: UIファイルを変更したら必ず更新する
// 形式: vYYYYMMDD-NN（同日複数回: NN=01,02,...）
export const APP_VERSION = 'v20260510-01'
```

- 言語・フレームワークが異なる場合も「1ファイル・1定数」の原則は同じ（例: `version.py` の `APP_VERSION`、`version.go` の `AppVersion` など）。
- パッケージマネージャの `package.json` `version` 等とは分けて考えてよいが、表示・運用上の正はこの定数とする。

---

## 2. 形式ルール

`vYYYYMMDD-NN`（**JST基準**）

| 部分 | 意味 | 例 |
|---|---|---|
| `v` | 固定プレフィックス | `v` |
| `YYYYMMDD` | JSTの日付 | `20260510` |
| `NN` | 同日内の連番（`01`, `02`, …） | `01` |

- 同じ日に複数回更新する場合は末尾 `NN` をインクリメント（`01` → `02` → …）。
- 基準タイムゾーンは **JST** で統一する。
- 例: 2026年5月10日の1回目 → `v20260510-01`、同日2回目 → `v20260510-02`。

---

## 3. 更新義務（破ると予期せぬルール）

- **UIに影響する変更を入れたら、必ず同じコミットで SSOT ファイルの値を更新する。**
- バージョン番号を更新せずに UI 影響ファイルを変更してはならない。
- 「修正が軽微だから」「急ぎだから」を理由に更新をスキップしない。

> UI影響の判断が曖昧な場合は、更新する側に倒す。動作中バージョンの追跡性を優先する。

---

## 4. 表示箇所（★アプリごとに確認して決定する）

**表示箇所はこのテンプレートでは固定しない。** 各アプリで以下を実装者がユーザーに確認のうえ決めること。

確認すべき事項:

- [ ] バージョン番号を画面上に表示するか（する / しない）
- [ ] 表示する場合、どの画面のどの位置に出すか（例: フッター右下、管理画面のみ 等）
- [ ] 複数画面に出す場合、どの画面に出すか

実装方針（表示する場合の展開ルール）:

- 表示は必ず SSOT の定数を `import` して行い、文字列をハードコードしない。
- 例: `import { APP_VERSION } from '.../version'` → 該当箇所で `{APP_VERSION}` を描画。

> 参考: Iconizer ではウィザード画面とアドオン画面のフッター右に表示している。ただしこれはアプリ固有の決定。他アプリでは別途ユーザーに確認すること。

---

## 5. 混同注意（別概念の "version"）

「バージョン番号（`APP_VERSION`）」と、機能上の別の "version" 文字列を混同しない。

- 例: 生成物のキャッシュバスティング用バージョン（`?v=...`）、APIバージョン、スキーマバージョン 等。
- これらはアプリ本体のバージョン番号とは無関係。命名で区別し、SSOTを共有しない。

---

## 6. 導入チェックリスト（新規アプリへ適用するとき）

- [ ] SSOTファイルを1つ作成し、定数を1つだけ export する
- [ ] 形式を `vYYYYMMDD-NN`（JST）に統一する
- [ ] 「UI変更時は必ず更新」を要約（CLAUDE.md等）に明記する
- [ ] 表示する／表示箇所をユーザーに確認する（第4節）
- [ ] 別概念の "version" と命名で区別する
